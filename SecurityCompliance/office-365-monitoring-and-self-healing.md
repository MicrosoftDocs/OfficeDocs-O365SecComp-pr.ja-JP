<span data-ttu-id="936a2-109">Office 365 の監視の基礎は、Azure、SQL Azure、および[オープンソースのストリーミングデータベーステクノロジ](http://cassandra.apache.org/)に基づいて構築されたデータ Insights エンジンを構成するテクノロジの集合体です。</span><span class="sxs-lookup"><span data-stu-id="936a2-109">Fundamental to Office 365 monitoring is a collection of technologies that comprise our Data Insights Engine, which is built on Azure, SQL Azure, and [open-source streaming database technology](http://cassandra.apache.org/).</span></span> <span data-ttu-id="936a2-110">データを収集して集約し、結論に達するように設計されています。</span><span class="sxs-lookup"><span data-stu-id="936a2-110">It is designed to collect and aggregate data and reach conclusions.</span></span> <span data-ttu-id="936a2-111">現時点では、多くの地域の複数のデータセンターに分散している10万台以上のサーバーから1時間あたり5億を超えるイベントを処理しています。これらの数は増加しています。</span><span class="sxs-lookup"><span data-stu-id="936a2-111">Currently, it processes more than 500 million events per hour from more than 100,000 servers (~15 TB per day) scattered across dozens of datacenters in many regions, and these numbers are growing.</span></span> 

<span data-ttu-id="936a2-112">Office 365 では、*外部の監視*を使用しています。これには、重要なすべてをテストする代理トランザクションの作成が含まれます。</span><span class="sxs-lookup"><span data-stu-id="936a2-112">Office 365 uses *outside-in monitoring*, which involves creating synthetic transactions to test everything that is important.</span></span> <span data-ttu-id="936a2-113">たとえば、Exchange Online では、各シナリオが5分ごとに散在しているすべてのデータベースをテストしており、システムに存在するすべてのものをほぼ継続的にカバーしています。</span><span class="sxs-lookup"><span data-stu-id="936a2-113">For example, in Exchange Online each scenario is testing every database worldwide every five minutes in a scattered fashion, providing near continuous coverage of everything that lives in the system.</span></span> <span data-ttu-id="936a2-114">複数の場所から、1日あたり2億5000万のテストトランザクションを実行して、サービスの堅牢なベースラインまたはハートビートを作成します。</span><span class="sxs-lookup"><span data-stu-id="936a2-114">From multiple locations, 250 million test transactions per day are performed to create a robust baseline or heartbeat for the service.</span></span>

<span data-ttu-id="936a2-120">この警告は、エラーアラートと赤のアラートの組み合わせに基づいて、問題を発生させる可能性のあるコンポーネントを示します。また、メールボックスサーバーを再起動することによって、システムがその問題を解決しようとしていることを示します。</span><span class="sxs-lookup"><span data-stu-id="936a2-120">Based on the combination of the failure alert and the Red Alerts, this alert indicates exactly which components could be having a problem, and that the system is going to try to correct the problem by itself by restarting a mailbox server.</span></span>

## <a name="autoreseed"></a><span data-ttu-id="936a2-137">AutoReseed</span><span class="sxs-lookup"><span data-stu-id="936a2-137">AutoReseed</span></span> 
<span data-ttu-id="936a2-138">Exchange Online サーバーは、複数のデータベースとそのログストリームを同じ非 RAID ディスクに格納する構成で展開されます。</span><span class="sxs-lookup"><span data-stu-id="936a2-138">Exchange Online servers are deployed in a configuration that stores multiple databases and their log streams on the same non-RAID disk.</span></span> <span data-ttu-id="936a2-139">この構成は多くの場合、ディスク上のデータを複製するために使用されている記憶域冗長メカニズム (RAID など) がないため、*多数のディスク*(JBOD) と呼ばれます。</span><span class="sxs-lookup"><span data-stu-id="936a2-139">This configuration is often referred to as *just a bunch of disks* (JBOD) because no storage redundancy mechanisms, such as RAID, are being used to duplicate the data on the disk.</span></span> <span data-ttu-id="936a2-140">ディスクが JBOD 環境で障害が発生すると、そのディスク上のデータは失われます。</span><span class="sxs-lookup"><span data-stu-id="936a2-140">When a disk fails in a JBOD environment, the data on that disk is lost.</span></span> 

<span data-ttu-id="936a2-141">exchange online のサイズと、その中に展開された it が数百万のディスクドライブになると、ディスクドライブの障害は exchange online で定期的に発生します。</span><span class="sxs-lookup"><span data-stu-id="936a2-141">Given the size of Exchange Online and the fact that deployed within it are millions of disk drives, disk drive failures are a regular occurrence in Exchange Online.</span></span> <span data-ttu-id="936a2-142">実際には、100を超える時間は毎日失敗します。</span><span class="sxs-lookup"><span data-stu-id="936a2-142">In fact, more than 100 fail every day.</span></span> <span data-ttu-id="936a2-143">オンプレミスのエンタープライズ展開でディスク障害が発生した場合、管理者は障害が発生したディスクを手動で交換して、影響を受けるデータを復元する必要があります。</span><span class="sxs-lookup"><span data-stu-id="936a2-143">When a disk fails in an on-premises enterprise deployment, an administrator must manually replace the failed disk and restore the affected data.</span></span> <span data-ttu-id="936a2-144">クラウド展開では、Office 365 のサイズ、オペレーター (クラウド管理者) による手動によるディスクの交換は現実的でも経済的でもありません。</span><span class="sxs-lookup"><span data-stu-id="936a2-144">In a cloud deployment the size of Office 365, having operators (cloud administrators) manually replacing disks is neither practical nor economically feasible.</span></span>
